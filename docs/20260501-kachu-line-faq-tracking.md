# Kachu LINE FAQ / Customer Message Capability Tracking

> 日期：2026-05-01
> 目的：記錄本日對 Kachu 是否具備顧客 LINE 訊息處理能力、是否應納入主線產品、以及後續執行順序的結論，供後續追蹤。

## 一、這次釐清的核心問題

本次討論聚焦於以下問題：

1. Kachu 目前是否真的有處理顧客 LINE 訊息的功能。
2. 先前對話中描述的「Line@ 客戶訊息助理 / PWA 工作台」與 Kachu 的相關性有多高。
3. 是否應把這個功能正式納入 Kachu 主線。
4. 如果要做，第一版應該執行哪些工作，而不是直接擴張成一個新產品。

## 二、本日核對過的事實

### 1. 程式碼層面

已確認以下程式碼存在：

- `src/kachu/line/webhook.py`
  - 會辨識老闆與非老闆的 LINE 訊息。
  - 對於非老闆訊息，會將文字存進 `ConversationTable`。
  - 之後會呼叫 `intent_router.dispatch(intent=Intent.FAQ_QUERY, ...)`。

- `src/kachu/tools/router.py`
  - 存在 `classify-message`
  - 存在 `retrieve-answer`
  - 存在 `generate-response`
  - 存在 `send-or-escalate`

這代表 Kachu repo 內確實已經有一套面向「顧客 LINE FAQ」的工具與入口草稿。

### 2. 文件層面

文件描述不一致，已確認如下：

- `README.md`
  - 沒有把顧客 LINE FAQ 作為清楚的正式產品功能來介紹。

- `Kachu-v2-Product-Plan.md`
  - 明確寫出 `Workflow 5: kachu_line_faq`。

- `docs/feature-comparison.md`
  - 將「顧客 LINE FAQ 自動回覆」標記為已實作。

- `docs/user-guide-boss.md`
  - 用老闆可理解的語氣寫到「回答顧客問題」。

### 3. 目前不能把它視為已完成產品功能的原因

雖然 repo 內有入口與工具，但本次檢視後的工作結論是：

- 尚未確認 AgentOS 那邊存在可正式執行的 `kachu_line_faq` workflow plan。
- README 沒有把它作為目前產品主線能力來說明。
- 目前整體產品設計與使用手冊的重心，仍是「老闆透過 LINE 操作 Kachu 來管理 Google / Meta / GA4 / 發文 /審批」。
- 顧客訊息能力沒有被完整驗證為一個穩定、可交付、可對外承諾的功能閉環。

因此，截至 2026-05-01，本能力較準確的判定是：

> Kachu repo 內已有顧客 LINE FAQ 的未完成基礎，但不能視為目前已成熟上線的正式產品功能。

## 三、與本次對話紀錄中產品方向的關係

本次外部對話描述的產品核心是：

1. 客戶透過 Line@ 傳訊息。
2. 系統讀取顧客訊息與顧客識別資訊。
3. AI 產生回覆建議。
4. 老闆看到草稿後確認或修改。
5. 系統再對顧客發送。
6. 系統逐步形成顧客記憶與後續跟進能力。

這與 Kachu 有高度產品相關性，因為 Kachu 本質上就是「微型創業者的 AI 數位營運幕僚」。

但是兩者目前仍有重要差別：

- 外部對話最後推進到「PWA 客服工作台」。
- Kachu 現行主架構則是「LINE 為老闆主要操作入口，Dashboard 是次要監控工具」。

所以如果把這個方向加進 Kachu，不應直接照抄成另一套 PWA 產品，而應優先補齊最符合 Kachu 原則的版本。

## 四、產品判斷：是否有必要納入 Kachu 主線

結論：有必要，但不應一次做成完整 PWA 客服系統。

理由如下：

1. Kachu 目前已經涵蓋內容生成、Google Business、GA4、審批、自動化提醒。
2. 但對微型創業家來說，「顧客來訊不漏接、回得快、回得像自己」是非常核心的日常營運問題。
3. 如果 Kachu 永遠只處理發文、評論、報表，而不碰顧客 Line@ 訊息，它比較像行銷助理，而不是完整的營運幕僚。
4. 因此，顧客訊息助理是 Kachu 長期產品方向中高度合理的一環。

不過，若直接做成 PWA 對話工作台，會產生以下問題：

1. 與 Kachu 現有「老闆不需要學新 App，只用 LINE 操作」的主張衝突。
2. 會大幅擴張產品範圍，從營運幕僚轉向客服系統。
3. 導入成本、教育成本、狀態同步與通知策略都會明顯變複雜。

因此，較佳的產品策略不是直接做 PWA，而是先補這個最小閉環：

> 顧客來訊 → AI 起草 → 老闆在 LINE 確認 / 修改 / 略過 → 系統回覆顧客。

## 五、建議執行範圍：第一版只做什麼

第一版只做「顧客文字 FAQ 回覆草稿」閉環，不做下列事項：

- 不做 PWA
- 不做完整 CRM
- 不做訂單管理
- 不做主動跟進提醒
- 不做跨平台客服整合
- 不做複雜客訴 / 退款 / 爭議處理自動化

第一版要完成的事情：

1. 顧客 LINE 訊息進來。
2. 系統識別顧客身分並保存最基本資料。
3. 系統根據知識庫與品牌語氣產生建議回覆。
4. 回覆不直接發送，而是先由老闆確認。
5. 老闆可以在 LINE 內發送、修改或略過。
6. 系統把這次顧客互動記錄下來，作為後續顧客記憶基礎。

## 六、建議執行順序

### Phase A：先補最小可用閉環

1. 正式定義此功能第一版只處理 FAQ 類顧客文字訊息。
2. 補資料模型與 repository，建立顧客識別與對話關聯。
3. 補 webhook 對顧客 profile / display name 的處理。
4. 在 AgentOS 補正式可執行的 `kachu_line_faq` workflow plan。
5. 將 FAQ 流程改成「老闆確認後才發送」，對齊 Kachu 的 approval 原則。
6. 補測試：顧客訊息進來、產草稿、老闆確認、成功發送、AI 無把握時升級。

### Phase B：有驗證結果後再做增強

1. 增加品牌化回答品質與知識庫命中率。
2. 增加輕量顧客記憶，例如常見問題、近期互動、是否需人工處理。
3. 加入風險策略：客訴 / 訂單 / 情緒性訊息只升級不自動建議。
4. 加入租戶開關，只對少數 beta 商家開放。

### Phase C：最後才考慮是否擴張為 PWA

只有在以下條件成立後，才考慮 PWA 客戶工作台：

1. 老闆真的頻繁使用此功能。
2. LINE 卡片確認流程開始出現操作瓶頸。
3. 有足夠證據顯示商家願意切換到一個新的客戶管理介面。

## 七、目前的決策結論

截至 2026-05-01，本次討論後的決策結論為：

1. Kachu 目前不能對外聲稱已完整具備顧客 LINE FAQ 產品能力。
2. 但此方向與 Kachu 的產品定位高度相關，應視為值得納入主線的中高優先級能力。
3. 第一版不應做 PWA，而應先做符合 Kachu 既有哲學的 LINE 內確認閉環。
4. 待第一版驗證成功後，再決定是否擴張成完整對話工作台或 PWA。

## 八、後續追蹤項目

後續實作或規劃時，應持續追蹤以下問題：

1. AgentOS 是否已補上正式的 `kachu_line_faq` workflow。
2. 顧客 profile 與 display name 的取得是否納入正式資料模型。
3. 顧客訊息的回覆是否已改為 approval-first，而不是直接發送。
4. README / 使用手冊 / 功能比較文件是否已與實際能力對齊。
5. 是否已有 beta 商家可以驗證「老闆是否願意直接用 LINE 卡片確認顧客回覆」。
